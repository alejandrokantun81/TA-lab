# Architecture

## 1. Architectural Intent and Scope

The primary objective of this architectural blueprint is to delineate a comprehensive, evaluated framework upon which the Locator, Connector, and Pathfinder agents shall be constructed. Building upon the discrete taxonomy knowledge graphs engineered during prior iterations (ESCO, O*NET, SFIA, BLS, and Lightcast), the present architecture establishes a unified backend topology. This paradigm facilitates sophisticated graph queries without necessitating the structural refactoring of any antecedent knowledge graph schemas.

The designated scope encompasses the agent topology, mapping the specific subagents, skills, and tools coordinated by the overarching orchestration layer. Furthermore, it prescribes the backend technological stack, detailing the requisite graph stores, vector indices, and language model providers. Explicitly excluded from the current operational phase are cross-taxonomy crosswalks, evaluator agents, and the definitive implementation of the agents themselves, which remain relegated to subsequent sprints.

## 2. User Journey and End-to-End Query Flow

The architectural design guarantees a seamless heuristic experience wherein end-users, operating within a natural-language chat interface, receive algorithmically grounded responses derived from complex taxonomy graphs, abstracted entirely from the underlying query mechanics.

**End-to-End Query Sequence:**
1. **Initialization:** The user submits a natural-language inquiry (e.g., "What requisite skills define a Data Analyst?").
2. **Orchestration:** The orchestration layer evaluates the semantic intent and delegates the computational task to the appropriate specialized agent (Locator, Connector, or Pathfinder).
3. **Resolution:** The Locator agent executes entity resolution, querying the vector index to map natural language to stable identifiers.
4. **Traversal:** The Connector agent utilizes the resolved identifiers to execute single-hop traversal queries within the designated knowledge graph, evaluating contiguous nodes.
5. **Synthesis:** The retrieved graph properties are synthesized into a coherent natural-language response and returned to the user interface.

## 3. Agent Topology and Delegation Mechanics

The agentic ecosystem adheres strictly to a hierarchical delegation paradigm, comprising autonomous agents that manage discrete, open-ended objectives. 

*   **Orchestration Layer:** Functions as the central cognitive router, determining the requisite agent based on isolated contextual demands.
*   **Locator Agent:** Entrusted with the critical task of entity resolution. It deploys a Resolution Subagent, loading taxonomy-specific skills to invoke fuzzy search mechanisms and stable-ID lookups against the vector index. All subsequent agents consume the resolved identifiers generated by the Locator.
*   **Connector Agent:** Engineered to explore nodal adjacencies. It employs a Traversal Subagent, utilizing taxonomy-specific skills to invoke single-hop Cypher queries and edge-property sorting mechanisms (e.g., evaluating O*NET importance metrics). Traversal logic defines a "neighbor" strictly via relationship types explicitly designated as traversable within the adapter configuration.
*   **Pathfinder Agent:** Designed for multi-hop directional routing, utilizing specialized tools to execute complex path-search algorithms across the knowledge graph.

## 4. Technological Infrastructure

The backend ecosystem comprises carefully selected frameworks designed to optimize the execution of the agentic topology. 

| Layer | Selected Infrastructure | Justification |
| :--- | :--- | :--- |
| **Graph Store** | Neo4j | Leverages existing Cypher-native taxonomy structures developed during initial modeling phases, ensuring immediate interoperability. |
| **Vector Store** | Supabase pgvector | Co-locates vector indices with relational metadata layers, circumscribing the infrastructure footprint to maintain architectural simplicity. |
| **Agent Framework** | LangGraph | Provides robust, native support for the Agent/Subagent delegation topologies, circumventing the necessity for custom orchestration loops. |
| **Model Provider** | Anthropic Claude (API) | Maintains alignment with pre-existing repository tooling references, pending comprehensive alternative evaluations. |

## 5. Taxonomy Ingestion and Adapter Interfaces

Given the structural heterogeneity of the source taxonomies (e.g., ESCO's direct edges versus SFIA's bridge-entity paradigms), the architecture implements a modular adapter pattern. Each taxonomy retains its foundational schema. A lightweight, taxonomy-specific adapter interfaces with the core orchestration layer, translating native relational labels into a unified programmatic interface. This methodology preserves the structural integrity of disparate graphs while enabling uniform querying mechanisms for the Locator and Connector agents.

## 6. Fundamental Constraints and Clarifications

*   **Entity Resolution Imperative:** Stable identifiers function as the paramount machine keys. Therefore, robust entity resolution constitutes a non-negotiable prerequisite, centralized exclusively within the Locator agent to prevent logic duplication.
*   **Implicit Cross-Domain Edges:** Relationships unrepresented in source files (e.g., O*NET Knowledge-to-Skills connections) must be systematically codified as explicit, auditable inferred edges, adhering strictly to the constitutional prohibition against silent algorithmic inference.
*   **Edge Weight Heterogeneity:** The architecture accommodates variable edge semantics (binary flags in ESCO versus scaled ratings in O*NET) by allowing the Connector agent to evaluate importance and level values as modifiable edge properties during traversal operations.
